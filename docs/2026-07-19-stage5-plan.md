## 1. Why Stage 5

Stage 4 made monitor **single-durable**. Remaining work is no longer dual-path migration; it is:

1. **Ops hygiene** — historical dead-letters, residual unexplained skips (zeta).
2. **Code surface reduction** — offline `generate_reply` / thin-monitor helpers still large.
3. **Reliability hardening** — CUA timeout class failures, provider transient fails.
4. **Product / multi-target rollout** — beyond canary targets with clear gates.

Without a named stage, these drift as ad-hoc tasks. Stage 5 is intentionally **hardening + cleanup**, not another transport rewrite.

## 2. Non-goals
