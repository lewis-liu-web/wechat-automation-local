## 1. Why Stage 5

Stage 4 made monitor **single-durable**. Remaining work:

1. **Ops hygiene** — archived dead-letters, zeta observation, CUA timeout class.
2. **Code surface honesty** — offline `generate_reply` stays but must not look “live”; thin-monitor UI must not imply dual-path.

4. **Ops gates clarity** — multi-target works; allowlist/opt-in/rollback must be visible and correct in UI.

Not a second transport migration.

### 5D. Multi-target ops gates (not “feature build”)

**Clarification:** Pipeline already multi-target. What remains is **guardrails**:

| Mechanism | Role today |
|-----------|------------|
| `targets[]` + monitor loop | N targets concurrent |
| `reliable_pipeline_target is True` | per-target durable opt-in / rollback |
| `dedicated_agent_instance_id` | pin job to instance |
| `test_target_only` + `test_target_names` | send allowlist (ops kill-switch) |

**Known production pair for E2E:**

| Target name (hosted) | Sender-side window title | Notes |
|----------------------|--------------------------|--------|
| `bot群聊测试` | `微信机器人测试` | Primary canary |
| `family` | `家人们` | Second multi-target canary |

**Add a production target checklist:**
1. Enable target in console; bind KB if needed.
2. Set `reliable_pipeline_target=true` (console: 可靠管线 opt-in).
3. Bind `dedicated_agent_instance_id` if required.
4. Set `cua_window_title` if hosted window title ≠ target name (sender remark window is for the **sender** account only).
5. If `test_target_only=true`, add exact `name` to `test_target_names` (runtime config) — **explicit approval**.
6. Stop monitor → change flags → start monitor (no live JSON race).
7. Smoke: sender window message → durable event → job → outbox sent/confirmed.
8. Rollback drill optional: opt-out → message advances without new durable row.

**Actions:**
- [x] Document checklist (this section).
- [x] UI shows opt-in + allowlist state (5F).
- [ ] Expand allowlist / disable `test_target_only` only with explicit approval.

**Exit:** Operators can add/opt-out a target without reading source. **Checklist met; expansion gated.**

## 6. Multi-target FAQ

**Q: 多 target 不是一直都支持吗？**  
**A: 是。** Monitor 按 `targets[]` 循环；durable 按 target 隔离；dedicated instance / allowlist 已是多 target 机制。  

Stage 5 不写“实现多 target”。只保证：
- UI/文档不说“只能单群”；
- 加群走 checklist；
- `test_target_only` 仍是**发送闸门**（现在 canary 名列表），不是“不支持多 target”。

**Q: family 怎么测？**  
Sender 账号打开备注为 **家人们** 的窗口发消息；托管侧 target name 仍是 **family**；需在 allowlist 且 `reliable_pipeline_target=true`。
